# 익명 리뷰 작성 및 신고·차단 설계

## 목표

Android와 Flutter/iOS에서 별도의 회원가입 화면 없이 책 리뷰를 작성할 수 있게 한다. 현재 두 앱이 사용하는 Firebase Anonymous Authentication의 UID를 내부 식별자로 사용하고, 작성·수정·삭제·신고·작성자 차단 경험을 두 플랫폼에서 동일하게 제공한다.

정상 리뷰는 즉시 공개하되 서버 검사에서 의심된 리뷰는 공개하지 않고 관리자 검토를 기다린다. Google Play와 Apple의 사용자 생성 콘텐츠 정책에 필요한 고지, 필터링, 신고, 차단, 운영자 조치를 함께 제공한다.

## 범위

이번 작업에 포함한다.

- Android와 Flutter/iOS의 리뷰 작성·수정·삭제 UI
- 자동 생성되며 작성 시 수정할 수 있는 익명 닉네임
- 리뷰 및 작성자 신고·차단
- 서버 측 입력 검사, 중복·도배 방지, 자동 숨김
- Vue 관리자 신고 검토 화면과 관리자 인증
- 웹 커뮤니티 가이드라인 페이지
- 리뷰 관련 Firestore 쓰기 경로 보호

웹에서는 리뷰를 계속 읽기 전용으로만 노출한다. 일반 회원가입, 로그인 UI, 공개 사용자 프로필, 댓글·답글, 좋아요는 이번 범위에 포함하지 않는다.

## 사용자 식별과 닉네임

두 앱은 기존 Firebase 익명 로그인으로 얻은 UID를 사용한다. 사용자에게 로그인이나 계정 생성 절차를 노출하지 않는다.

새 리뷰 작성 시 기본 닉네임은 UID를 그대로 노출하지 않고 해시한 값으로 `익명의 독자 A3F2` 형태로 만든다. 사용자는 작성창에서 닉네임을 수정할 수 있다. 닉네임과 본문에는 금칙어, URL, 이메일, 전화번호 등 개인정보·홍보 패턴 검사를 동일하게 적용한다.

`user_uid`와 작성 시점의 `user_name`은 각 리뷰 문서에 함께 저장한다. 사용자 컬렉션을 매번 조회하거나 조인하지 않는다. 사용자가 기존 리뷰를 편집하면서 닉네임을 바꾸면 그 리뷰의 `user_name`만 갱신한다. 다른 과거 리뷰의 이름을 일괄 변경하지 않는다. 앱은 마지막으로 사용한 닉네임을 로컬에 저장하여 다음 리뷰의 기본값으로 재사용할 수 있다.

이 방식은 익명 사용자 단계에서 읽기 비용과 동기화 복잡도를 낮춘다. 향후 정식 계정과 공개 프로필을 도입할 때 전역 닉네임 정책과 기존 리뷰 마이그레이션을 별도로 설계한다.

## 데이터 모델

### `book_reviews/{reviewId}`

한 사용자는 한 책에 리뷰 하나만 작성할 수 있다. 문서 ID는 서버에서 `book_id`와 `user_uid`를 이용해 결정적으로 생성하여 중복 작성을 막는다.

- `book_id`: 책 문서 ID
- `user_uid`: Firebase 익명 UID
- `user_name`: 리뷰에 표시할 닉네임
- `review`: 리뷰 본문
- `rating`: 정수 별점
- `created_at`: 서버 Timestamp
- `updated_at`: 서버 Timestamp
- `hide`: 기존 호환 필드. 숨김이면 문자열 `"1"`, 공개 상태에서는 필드를 생략하거나 기존 공개값을 유지
- `moderation_status`: `published | pending | hidden`
- `report_count`: 서로 다른 신고자의 유효 신고 수
- `policy_version`: 작성 시 동의한 커뮤니티 정책 버전

기존 Android, Flutter/iOS, SSR 웹, Vue 관리자가 공유하는 `hide: "1"` 규약은 유지한다. 과거 리뷰에는 새 필드가 없을 수 있으므로 모든 조회·표시 코드는 누락 값을 허용한다.

### `review_reports/{reportId}`

문서 ID는 `review_id`와 `reporter_uid`로 결정적으로 생성하여 한 사용자의 중복 신고를 한 번으로 제한한다.

- `review_id`, `book_id`
- `review_author_uid`
- `reporter_uid`
- `reason`: `spam | offensive | harassment | inappropriate | other`
- `detail`: 선택 입력
- `created_at`: 서버 Timestamp
- `status`: `open | reviewed | dismissed`

### 운영 데이터

차단 사용자와 운영 제재는 별도 내부 컬렉션에 저장한다.

- 앱의 작성자 차단 목록은 우선 각 기기의 로컬 저장소에 UID 목록으로 보관한다.
- 서버의 `review_user_bans/{uid}`는 운영자가 리뷰 작성 금지한 UID와 사유·시각을 저장한다.
- 관리자 SDK만 신고 및 운영 내부 컬렉션을 읽고 쓸 수 있다.

## 작성·수정·삭제 흐름

책 상세의 `독자 리뷰` 제목 오른쪽에 `리뷰쓰기` 버튼을 둔다.

1. 앱은 익명 인증이 완료됐는지 확인한다.
2. 최초 작성 또는 정책 버전 변경 후 첫 작성이면 공개 리뷰 고지와 커뮤니티 가이드라인 동의를 받는다.
3. 작성창은 별점, 닉네임, 본문을 받는다. 닉네임에는 자동값 또는 마지막 사용값을 채운다.
4. 앱은 Cloud Function에 제출하고 Firestore에 직접 쓰지 않는다.
5. 서버는 인증, 소유권, 길이, 별점, 금칙어·연락처·URL, 작성 빈도, 운영 제재를 검사한다.
6. 정상 리뷰는 `published`로 즉시 공개한다. 의심 리뷰는 `pending`과 `hide: "1"`로 저장하고 사용자에게 검토 대기임을 알린다. 명백히 허용되지 않는 입력은 저장하지 않고 수정 안내를 반환한다.

자신의 리뷰 카드에는 수정과 삭제 메뉴를 제공한다. 앱이 보내는 현재 Firebase 익명 UID와 리뷰의 `user_uid`가 일치할 때만 서버가 수정·삭제를 허용한다. 수정은 작성과 같은 서버 검사를 다시 거치며, 닉네임도 해당 리뷰에서 수정할 수 있다. 앱 삭제나 재설치로 익명 UID가 바뀐 경우에는 이전 리뷰의 소유권을 복구하지 않는다.

## 신고와 차단

다른 사용자의 리뷰 카드에는 `신고`와 `작성자 차단`을 제공한다.

- 신고 사유를 선택하여 Cloud Function으로 제출한다.
- 같은 UID가 같은 리뷰를 여러 번 신고해도 한 번만 계산한다.
- 유효 신고가 1~2건이면 관리자 신고 목록에 표시하고 리뷰는 공개 상태를 유지한다.
- 서로 다른 UID의 유효 신고가 3건이면 리뷰를 즉시 `hidden`, `hide: "1"`로 전환한다.
- 관리자는 숨김 유지, 문제없음 복구, 신고 기각, 작성자 작성 금지를 처리할 수 있다.
- 문제없음으로 복구된 신고 묶음은 종결하여 동일 신고로 다시 자동 숨김되지 않게 한다.

작성자 차단은 신고와 별개다. 사용자가 차단하면 해당 UID의 리뷰를 현재 기기에서 즉시 모두 숨긴다. 신고하지 않고 차단만 할 수도 있다. 익명 UID가 앱 데이터 삭제나 재설치로 달라질 수 있다는 한계는 이번 비회원 버전에서 수용한다.

## 커뮤니티 정책

Next.js 공개 웹에 `https://bookchelin.com/community-guidelines` 페이지를 추가한다. 금지 콘텐츠, 리뷰 공개 범위, 신고·숨김·제재 가능성, 문의 연락처를 명시한다. 웹에 두어 앱 배포 없이 내용을 수정할 수 있게 한다.

앱의 리뷰 작성 화면에는 핵심 고지와 정책 링크를 표시한다. 링크는 앱 안의 웹뷰 또는 플랫폼 커스텀 탭으로 연다. 앱은 동의한 `policy_version`을 로컬에 저장하고 리뷰에도 기록한다. 정책 버전이 바뀌면 다음 작성 시 다시 동의를 받는다.

## 서버와 보안 경계

새 리뷰 API는 기존 프로젝트 관례에 맞춰 Cloud Functions v1으로 구현한다. v2 함수를 섞지 않는다.

- 리뷰 작성·수정·삭제
- 리뷰 신고
- 관리자 검토·복구·숨김·사용자 제재

클라이언트의 `book_reviews` 및 `review_reports` 직접 쓰기는 Firestore Rules에서 거부하고 Cloud Functions Admin SDK만 쓰게 한다. 공개 리뷰 읽기는 유지한다. 신고·운영 데이터는 일반 클라이언트가 직접 읽을 수 없다.

현재 전체 허용 규칙을 한 번에 전면 잠그지 않는다. 리뷰 관련 컬렉션을 우선 명시적으로 보호하되, 기존의 광범위 허용 규칙이 이를 다시 허용하지 않도록 규칙 구조를 테스트한다. 배포 전에 Android, Flutter/iOS, SSR 웹, Vue 관리자에서 사용하는 모든 리뷰 읽기·쓰기 경로를 확인한다.

Vue 관리자는 현재의 익명 인증으로 운영 작업을 계속할 수 없다. 허용된 관리자 이메일 계정으로 로그인하고 서버에서 관리자 권한을 확인한 경우에만 신고 원문과 운영 API에 접근할 수 있게 한다.

## 관리자 기능

Vue 관리자에 신고 검토 목록을 추가한다.

- 최근 신고와 상태 필터
- 책, 리뷰 원문, 작성 닉네임, 작성자 UID의 축약 표시, 신고 사유와 고유 신고자 수
- 리뷰 숨김·복구
- 신고 문제없음 처리
- 작성자 리뷰 작성 금지·해제

기존 리뷰 관리 화면의 `hide: "1"` 토글과 충돌하지 않게 동일한 서버 운영 API를 사용한다. 자동 필터로 `pending`이 된 리뷰도 같은 검토 화면에서 처리한다.

## 오류 처리

- 인증이 끊기면 익명 재인증 후 한 번 재시도한다.
- 네트워크 실패 시 작성 내용을 화면에 유지하고 재시도할 수 있게 한다.
- 중복 제출은 결정적 문서 ID와 서버 트랜잭션으로 한 건만 반영한다.
- 이미 삭제되거나 숨겨진 리뷰 신고는 안전하게 종료하고 사용자에게 현재 상태를 알린다.
- 앱에는 구체적인 필터 규칙이나 내부 제재 정보 대신 수정 가능한 사용자 메시지를 반환한다.

## 테스트와 검증

- 서버 단위 테스트: 닉네임 생성, 입력 검사, 소유권, 한 책 한 리뷰, 신고 중복 방지, 3건 자동 숨김, 복구 후 재숨김 방지, 제재 사용자 차단
- Firestore Rules 에뮬레이터 테스트: 공개 읽기, 리뷰·신고 직접 쓰기 거부, 내부 운영 데이터 접근 거부
- Android와 Flutter 위젯/로직 테스트: 작성·수정·삭제, 정책 동의, 신고, 로컬 차단, 과거 필드 누락 리뷰 표시
- 통합 확인: Android·iOS에서 작성한 리뷰가 양쪽 앱과 SSR 웹에서 동일하게 보이고 `hide: "1"` 리뷰는 모두 숨겨지는지 확인
- 관리자 확인: 신고 검토, 자동 숨김, 복구, 사용자 제재 및 권한 없는 접근 거부

## 출시 순서

1. 서버 함수, 보안 규칙, 커뮤니티 정책 페이지, 관리자 검토 기능을 먼저 배포한다.
2. 운영 흐름을 점검한 뒤 Android와 Flutter/iOS 앱을 배포한다.
3. 초기에는 신고 목록을 주 1~2회 확인하고 자동 필터 오탐과 신고 임계치 3건의 적절성을 관찰한다.
4. 운영 데이터가 쌓이면 임계치나 필터 규칙을 서버 설정으로 조정한다.
